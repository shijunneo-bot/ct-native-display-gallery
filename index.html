How Native Display works (Web vs App)Start here›

Imagine your app is a shop. A pop-up is someone stepping in front of a customer to wave a flyer. Native Display is different: it quietly changes what sits on the shelf, a card in the home feed, a banner on a page, a strip at checkout. Customers never feel interrupted, so they do not swat it away.

Why this matters for money: the home feed and checkout are the busiest spots in the whole product. Native Display puts the right offer in front of the right person exactly there, and it is already part of your plan on Advanced and Cutting Edge. No extra spend, just campaigns you have not run yet.

App Native Display Included · Advanced+

The CleverTap SDK hands your app a Display Unit (title, message, media, icon, key-values). Your app decides where and how it renders. iOS SDK 3.7.2+ and Android SDK 3.6.2+.

Templates: Simple Message, Carousel Message (up to 5 slides), Message with Icon, Custom Key-Value, Image-only Simple, Image-only Carousel.

Web Native Display Included · Advanced+

CleverTap either renders into a Div you nominate (banners, carousels) or fires a browser event your site listens for (key-value, JSON). Needs the current Web SDK (1.3.0+; Visual Editor needs 1.13.1+).

Templates: Custom Key-Value, Banner, Banner Carousel (up to 7 rotations), Banner with Text, Banner Carousel with Text. Add-on unlocks Custom HTML and Custom JSON templates plus the Visual Editor.

Who renders what

Rendering handled by CleverTapRendering handled by your code
Web Banners, Banner Carousels, Custom HTML, Visual Editor element changes. CleverTap injects into your target selector.All App templates (the app builds the view from the Display Unit), Web Key-Value and JSON campaigns (a browser event delivers the payload, your site renders it).

Priority: on web, priority is keyed to the CSS selector; when several campaigns target the same element only the highest priority runs, and a campaign runs entirely or not at all. On app, campaign priority runs 1 to 10 (10 wins); newer SDKs support multiple ND campaigns from a single event for multi-slot layouts. The same event can also feed different campaigns through different event property filters: one Charged event, filtered on Category = Fashion vs Category = Grocery, can light two different slots with two different messages.

Fallbacks are not optional. Always configure default content in the slot. If CleverTap's response is delayed or empty, the page or app should show its own default rather than a hole. On web, personalization uses user properties from the current session only; properties written via the server API are not available client-side.
Key-Value Pair execution (Web + App)Dev pattern›

Key-Value is a simple handshake: the marketer sends the ingredients (a name, an offer, a price), and the app's own code does the cooking, drawing the card in the app's own style. Use it when the message must look like it was born inside the product. The payoff: the marketing team can change what the card says any day of the week, without asking engineering to touch the app.

Web: the CustomEvent pattern

CleverTap fires a browser event named CT_web_native_display. The first key in the payload is always topic (the marketer sets its value), which routes content to the right slot.

// 1. Listen once, route by topic
document.addEventListener("CT_web_native_display", function(e){
  const data = e.detail;            // { kv:{ topic, ... }, msgId, wzrk_pivot }
  if (data.kv.topic === "cart_nudge") {
    renderCartNudge(data.kv);        // your render into the slot
    clevertap.renderNotificationViewed(data);   // 2. raise Viewed
  }
});
// 3. On click of the rendered element:
// clevertap.renderNotificationClicked(data);

Related events for other web templates: CT_web_native_display_builder (Visual Editor JSON campaigns) and CT_web_native_display_json (JSON template). Same-named events are grouped; only the highest-priority campaign fires.

App: the Display Unit pattern

// Android: receive units, read the KV map, render, raise events
class MainActivity : DisplayUnitListener {
  override fun onDisplayUnitsLoaded(units: ArrayList<CleverTapDisplayUnit>) {
    val kv = units[0].customExtras       // dashboard key-values
    renderHomeCard(kv)
    ct.pushDisplayUnitViewedEventForID(units[0].unitID)
  }
}
// On tap: ct.pushDisplayUnitClickedEventForID(unitID)
// iOS mirror: CleverTapDisplayUnitDelegate + getAllDisplayUnits()
Stats live or die on these calls. Viewed and Clicked on the dashboard come only from your code raising them. Miss the calls and the campaign reports zero even while rendering perfectly. In list views (RecyclerView and friends), Viewed fires each time the unit scrolls into the viewport, so expect multiple Viewed per session.
Developer setup (App ND · Web ND · Visual Editor)Handoff-ready›

App Native Display setup

Three steps. First, register a listener so the app receives units when a campaign triggers: on Android set a DisplayUnitListener and read units in onDisplayUnitsLoaded, or fetch on demand with getAllDisplayUnits() / getDisplayUnitForID(unitID); on iOS implement CleverTapDisplayUnitDelegate and call setDisplayUnitDelegate. Second, render your own view from the unit's fields (title, message, media, icon) and, for Custom Key-Value, from customExtras. Third, tell CleverTap what happened: raise Viewed when the unit enters the viewport and Clicked on tap, both by unit ID. React Native, Flutter, and Unity have mirror APIs.

Web Native Display setup

Integrate the current Web SDK (account id, region, token) and turn on personalization with clevertap.enablePersonalization = true. For Banner and Carousel templates, nominate a Div ID at campaign time; CleverTap replaces that container's content. If your container has a fixed CSS height, set the same Div height in the dashboard; with no height set, the banner takes 100% width with auto height by aspect ratio. Plan for the session rule: client-side personalization sees only properties set in the current session.

Visual Editor setup (Advanced Web Personalization add-on)

The Visual Editor is part of the paid Advanced Web Personalization add-on and is in Private Beta, so it needs enabling on the account by your CleverTap contact. Prerequisites: Web SDK 1.13.1+, a Content Security Policy whose frame-ancestors includes self, and headers Cross-Origin-Opener-Policy: same-origin-allow-popups plus Cross-Origin-Embedder-Policy: unsafe-none. Build flow: open the Web ND builder, choose Advanced Personalization, enter the page URL, and the live page loads in the editor. Point-and-click an element to restyle it, or switch to Edit HTML / Add JSON; use liquid {{ }} for profile and event personalization; switch on element-level Click Tracking where you want click analytics. Preview in a new tab, Save, then publish. Work on staging first, verify, then publish to production.

Visual Editor add-on consideration + Custom Templates breakdownAdd-on›

Plain version: standard Web Native Display lets you fill the shelves you agreed on in advance. The Visual Editor add-on lets you redecorate the whole shop window. It is for changing how the website looks: text, images, buttons, colors, any element on the page, without waiting for the web team's next release.

Why this matters for money: every week a better headline or button sits in a backlog, it earns nothing. The Visual Editor turns "we'll ship it next sprint" into "it is live this afternoon", so winning ideas start paying sooner and losing ideas get switched off faster.

Visual Editor Add-on · Private Beta

Loads the live page into an editor. Point, click, and change text, images, CTAs, colors, spacing, or any HTML element, then target by segment and publish. Includes a Basic (Form) mode for quick swaps, liquid personalization, element-level click tracking, device toggle, changelog, and multi-page support.

Best for: hero and CTA iteration, seasonal reskins, segment-specific page variants, testing copy without a web release cycle.

Custom Templates Add-on

Custom HTML Template: write HTML/CSS/JS in a built-in editor for layout changes, embedded recommendations, reordering, and personalized offer blocks. CleverTap injects it into the target selector.

Custom JSON Template: sends structured JSON to the site; your front end renders it. The right choice when the design system must stay fully in your codebase but the content should be campaign-controlled.

Commercial framing that tends to land: the subscription already includes the shelf (Banner, Carousel, Key-Value). The add-on buys the ability to rearrange the store per shopper, without waiting for the next release train.

The honest limitations list

  • Website only. It does not touch iOS or Android app screens. App-side changes are Native Display (content) or Product Experiences (behavior).
  • It edits pages that exist. Not a landing-page builder: you give it a URL, it redecorates that page. New pages come from your web team, then VE personalizes them.
  • Looks, not logic. It changes what the page shows, not what the server computes. Real price logic, checkout rules, and app behavior belong to Product Experiences or your backend.
  • Setup prerequisites. Web SDK 1.13.1+, CSP frame-ancestors including self, and COOP/COEP headers. Budget a small web-team ticket for this, once.
  • Private Beta. Enabled per account by CleverTap, tested first on Shopify. Confirm support for the client's stack before promising a date.
  • Session-scope personalization. Client-side liquid sees properties set in the current session; server-API-written properties are not available in the browser.
  • Forms: visible half only. You can inject a form's markup, but submissions route to your site's endpoint; CleverTap does not process them.
  • Publish gate and priority. Nothing reaches live users until Publish, and when campaigns collide on one selector, only the highest priority runs.
  • Dynamic SPAs need care. Heavily client-rendered pages need stable selectors for edits to stick; verify behavior on the client's framework during the beta.
The extent, in one line: if it is visible on the website and has a stable place in the page, the Visual Editor can probably change it per segment without a release. If it is a new page, an app screen, or server logic, it is the wrong tool, and saying so early is what keeps the add-on pitch credible.

Visual Editor vs Product Experiences: which tool, which jobDecision table›

The simplest way to hold the two add-ons in your head: Visual Editor changes how it looks. Product Experiences changes how it works. One repaints the website (colors, words, images, buttons). The other rewires the app (features on or off, settings, whole flows). Both go live from the dashboard, and neither needs the app submitted to the App Store or Play Store for review, so a change that used to wait weeks for a release can ship today.

FAQ: the 20 questions clients actually askCall-ready›
How do I choose where Native Display appears in the app?

Placement is decided in the app's code, once. Your developers pick the slots (home feed position 2, checkout strip, account banner) and render Display Units there. After that, marketers fill those slots from the dashboard without touching code. Best practice: agree a named slot list with the product team (like agreeing shelf locations in a store), and pass the slot name in the campaign's key-value pairs so the app knows which shelf each campaign belongs on.

How do I choose where it appears on the website?

Two ways. For Banner and Carousel templates, you enter the Div ID of the target container when building the campaign, and CleverTap renders into that box. For Key-Value campaigns, the first key is always topic: your site listens for the event, reads the topic, and routes the content to the right spot. With the Visual Editor add-on, you skip both and simply click the element on a live view of the page.

What is the difference between a default template and Custom Key-Value?

A default template (Simple Message, Carousel, Banner) is a ready-made frame: CleverTap defines the structure, you fill in text and images, and it renders with no design work. Custom Key-Value sends only the raw ingredients (names, prices, offer text) and your own code draws the card, so it can look exactly like the product. Templates are faster to launch; Key-Value gives total design freedom in exchange for one-time developer work.

How do I actually use custom key-value pairs, step by step?

Four steps. One: marketer adds key-value pairs in the campaign (on web, the first key is topic, which acts like an address label). Two: the developer adds a listener, on web for the CT_web_native_display event, on app via the Display Unit callback. Three: the code reads the values and draws the element. Four: the code reports back by raising Notification Viewed and Notification Clicked. Steps two to four are built once; after that, marketers change the content any day without engineering.

Do I need developers for every campaign?

No. Developers are needed once per surface: to set up the SDK, choose app slots, and (for Key-Value) build the render-and-report code. After that one-time wiring, template campaigns and content changes are fully marketer-run. The recurring engineering cost is close to zero, which is a big part of the commercial case.

Is Native Display included in my plan or an extra cost?

Web and App Native Display are included on Advanced and Cutting Edge plans. On Essentials, App Native Display is an add-on and Web Native Display is not offered. So for most enterprise accounts this is capability already paid for and often under-used. (Confirm your exact contract line items with your CleverTap team.)

How is this different from in-app messages and pop-ups?

Pop-ups interrupt: they appear over the screen and demand a decision. Native Display sits inside the screen, as part of the feed or page, so nobody dismisses it in annoyance. Practical effect: you can run it always-on in high-traffic spots without hurting the experience, which pop-up fatigue does not allow.

Can I personalize with the user's name, points, or prices?

Yes. Any field marked with the @ symbol accepts personalization: type @ or {{ }} and pick a profile or event property, with a default fallback value. That covers names, balances, dates, amounts, even the image URL and the deep link. One web caveat: client-side personalization only sees properties set in the current session, not values written through the server API.

Can it show product recommendations automatically?

Yes, with a catalog. Upload your product catalog, connect it to an event, and the recommendation engine can fill a carousel per user. Plan for the warm-up: relationships form from your data over roughly the first 24 hours after upload, and published recommendations can take up to 24 hours to go live. Not a same-morning launch, but fully automatic after that. One plan note: Recommendations with Catalog 2.0 sit on the Cutting Edge plan only, so check the client’s tier before promising the carousel.

Why are my campaign stats showing zero even though the unit renders?

Almost always because the code is not raising the events. Viewed and Clicked on the dashboard come only from the app or site calling the report functions. If those calls are missing, the campaign works perfectly and reports nothing. It is the first thing to check, before anything else.

Can I A/B test Native Display campaigns?

Yes: up to 3 message variants in an A/B test, plus Split Delivery, plus variation by user property (up to 50 variants), plus control groups to measure true lift. So you can prove the revenue effect of a slot, not just report clicks.

What happens if two campaigns target the same slot?

Priority decides. On web, priority is keyed to the target selector and only the highest-priority campaign runs (a campaign runs entirely or not at all). On app, campaign priority runs 1 to 10 and 10 wins; newer SDKs also allow several campaigns from one event to fill several slots at once. And one underused trick: the same event can drive different campaigns via different event property filters, so Charged with Category = Fashion and Charged with Category = Grocery can each trigger their own unit.

What media can I use?

Images (jpg, jpeg, png, gif), audio (mp3) and video (mp4) on app templates; note that png converts to jpeg using your chosen background color. App carousels hold up to 5 slides; web banner carousels rotate up to 7 banners.

What if CleverTap is slow or sends nothing? Will there be a hole in my page?

Only if you skip the fallback, so never skip the fallback. Always configure default content in the slot: the page or app shows its own default until CleverTap's content arrives, and keeps showing it if nothing arrives. An empty slot is worse than no campaign.

Which SDK versions do we need?

App: iOS 3.7.2+ and Android 3.6.2+ for Native Display basics (Linked Content in units needs much newer builds). Web: the current Web SDK, 1.3.0+ for the feature and 1.13.1+ if you want the Visual Editor. Worth checking early, because an SDK upgrade is the one item with a real lead time.

Can I target by segment and live behavior?

Yes, the full CleverTap targeting toolkit applies: saved segments, user properties, and live behavior for triggered campaigns, plus scheduling and delivery preferences. Native Display is a channel like any other; only the rendering surface is different.

What does the Visual Editor add that I do not already have?

Standard Web Native Display fills slots you agreed in advance. The Visual Editor (part of the paid Advanced Web Personalization add-on) lets a marketer click any element on a live view of the page and change it: text, images, buttons, colors, spacing. It removes the two remaining bottlenecks: pre-agreed slots and the web release cycle.

Can the Visual Editor create a custom landing page from scratch?

Honest answer: no. The Visual Editor edits pages that already exist; you give it a URL and it loads that live page for editing. It is a redecorating tool, not a page builder. If you need a genuinely new landing page, your web team (or a page-builder tool) creates it, and then the Visual Editor can personalize every element of it per segment, which is where it shines.

Can I embed a form (like an email sign-up) through the Visual Editor?

Partly, with care. The editor's Edit HTML and Insert Element features can inject an HTML block, and the Custom HTML template can carry a form's markup. But CleverTap does not process the form submission: where the entered data goes must be wired by your web team (their endpoint, their validation). So treat it as "yes for the visible form, your site for the plumbing", and verify the specifics with your CleverTap contact since the Visual Editor is in Private Beta and capabilities are still moving.

Will Native Display slow my app or site down?

It is designed not to: the content request runs alongside your page or app load, and with a fallback configured the user never waits on it. The practical safeguard is the same as the previous answer: default content in the slot, so nothing blocks and nothing gapes.

Where should we start if we have never used it?

One slot, one vertical use case, two weeks. Pick the busiest screen (usually the home feed or the cart), run one templated campaign with a control group, and read the conversion lift, not the clicks. That single number is what funds the second slot, and eventually the add-on conversation.
